name: note-to-techbook
description: Note.com からエクスポートした Markdown ファイルを techbook-template 記法に変換する。単一ファイルまたは一括変換に対応し、frontmatter 整理・広告ブロック削除・記法変換を行う。
---

Note.com からエクスポートした Markdown ファイルを techbook-template 記法に変換します。
以下の3フェーズで作業を進めてください。

---

## フェーズ1: 対象ファイルと変換情報の確認

ユーザーに以下を確認してください:

1. **変換モード**
   - `単一ファイル`: 指定ファイル1件を変換
   - `一括`: `backup/` 以下の全ファイルを対象に変換

2. **変換対象ファイルパス**（単一モードの場合）

3. **guid → docs ファイル名の対応表**（`{{chapref}}` 変換に必要）
   - Note.com の記事 URL には guid（例: `n1a2b3c4d5e6`）が含まれる
   - 対応する `docs/` ファイル名（例: `01-introduction`）を確認する
   - 不明な guid は `TODO: chapref` コメントとして残す

一括モードの場合: `backup/` ディレクトリの内容を確認してファイル一覧を提示する。

---

## フェーズ2: 変換の実施

対象ファイルを読み込み、以下の順で変換を適用してください。

### A. Frontmatter 整理

**除去するキー:**

```
title, pubDate, modifiedDate, guid, link, creator, status, draft
```

**修正パターン:**

- `draft: false---` のように壊れた frontmatter（`---` が同一行に混入）は修正する
- frontmatter ブロック全体が壊れている場合は整形し直す

**`description` / `description1` / `description2` の処理:**

- 本文冒頭（H1・サブタイトル・空行を除いた最初の段落）から説明文を自動生成する
- `description1`: 章カバー上部に表示する短いラベル（例: `ComfyUI編`、`SAA編` など章の種別）
- `description2`: 章カバー下部に表示する詳細説明文（説明文の全文）
- 生成した各フィールドをユーザーに提示して確認を取る

**テーマカラーの選択:**

変換時にその章のテーマカラーをユーザーに選んでもらう。
以下の選択肢を提示し、章の内容・雰囲気に合うものを選択させる:

| カラー名 | 変数名 | イメージ |
|---------|--------|---------|
| blue（デフォルト） | `--tw-blue-*` | 技術・汎用 |
| sky | `--tw-sky-*` | 軽やか・クラウド |
| cyan | `--tw-cyan-*` | デジタル・清涼 |
| indigo | `--tw-indigo-*` | 知性・深み |
| violet | `--tw-violet-*` | 創造・AI |
| purple | `--tw-purple-*` | 独創・魔法 |
| fuchsia | `--tw-fuchsia-*` | 鮮やか・個性 |
| pink | `--tw-pink-*` | 柔らか・可愛い |
| rose | `--tw-rose-*` | 情熱・愛 |
| red | `--tw-red-*` | 警告・強調 |
| orange | `--tw-orange-*` | エネルギー・活発 |
| amber | `--tw-amber-*` | 温かみ・注意 |
| yellow | `--tw-yellow-*` | 明るさ・アイデア |
| lime | `--tw-lime-*` | 新鮮・軽快 |
| green | `--tw-green-*` | 安定・自然 |
| emerald | `--tw-emerald-*` | 成功・バランス |
| teal | `--tw-teal-*` | 落ち着き・調和 |
| slate | `--tw-slate-*` | クール・モノトーン |

選択後、frontmatter の `color.primary` ブロックを以下の形式で設定する（例: `violet` を選んだ場合）:

```yaml
color:
  primary:
    "500": "var(--tw-violet-500)"
    "400": "var(--tw-violet-400)"
    "200": "var(--tw-violet-200)"
    "50": "var(--tw-violet-50)"
```

既存の `color` ブロックがある場合は上書きする（ユーザーが「変更しない」を選んだ場合はそのまま）。

**変換後の frontmatter 形式:**

```markdown
---
description1: （短いラベル、例: ComfyUI編）
description2: >-
  （確認済みの詳細説明文）
color:
  primary:
    "500": "var(--tw-{color}-500)"
    "400": "var(--tw-{color}-400)"
    "200": "var(--tw-{color}-200)"
    "50": "var(--tw-{color}-50)"
columns: false
---
```

`columns` が元々ない場合は追加しない。

### B. コンテンツクリーンアップ

以下のパターンを**コードブロック外のみ**対象に削除・整理する。

**① 広告 blockquote の削除**

以下のいずれかで始まる blockquote ブロック全体を削除する:

```
> 🎁
> 【今だけ無料】
```

ブロック内の複数行（連続した `>` 行）もまとめて削除する。

**② Note.com スタンドアロン URL リンクの削除**

1行全体が Note.com へのリンクである行を削除する:

```
[https://note.com/...](https://note.com/...)
```

条件: 行頭から始まり、行末がリンクで終わる（前後に他テキストがない）。

**③ マガジン宣伝文の削除**

以下のキーワードを含む段落・文を削除する:

- 「このマガジン」
- 「マガジンに収録」
- 「マガジンの購読」
- 「マガジンへの追加」

段落単位（空行で区切られたブロック）で削除する。

**④ 絵文字の除去**

コードブロック外のテキストから絵文字を除去する。除去後に不自然なスペース（連続スペース・行頭スペース）が残る場合は整理する。

### C. 記法変換

**① 区切り線 → 改ページ**

```
* * *
```
```
-----
```

→ `{{page-break}}` に変換する。

**② コードブロックへのタイトル追加**

タイトルのないコードブロックに `<!-- title: タイトル -->` を追加する:

```markdown
<!-- title: ファイル名や処理内容 -->
```js
// コード
```
```

- タイトルは前後の文脈（見出し・説明文）から適切に推測して付ける
- すでに `<!-- title: ... -->` または `title:` が付いているブロックは変更しない

**③ コードブロックへの言語設定**

言語指定のない開始行（` ``` ` のみ）に言語を追加する。**閉じる側の ` ``` ` は変更しない。**

| 内容 | 言語 |
|------|------|
| シェルコマンド（bash/brew/git/curl/ssh など） | `bash` |
| ターミナル出力・エラーログ・成功メッセージ | `text` |
| AI画像生成プロンプト（LoRA・キャラ一覧含む） | `text` |
| JSON設定フラグメント | `json` |
| URL のみ | `text` |

**注意:** プロンプト例が多い章（キャラクター一覧・LoRA設定など）は言語なしブロックが多数あるため見落としに注意する。変換後に ` ``` ` のみの開始行が残っていないか必ず確認する。

**③ Note.com 内部リンク → chapref**

URL パターン: `https://note.com/.../n/{guid}`

- guid が対応表にある場合: `{{chapref 'ファイル名（拡張子なし）'}}`
- guid が対応表にない場合: `TODO: chapref '{guid}'`（コメントとして残す）

例:

```markdown
[第1章を参照](https://note.com/user/n/n1a2b3c4d5e6)
↓
{{chapref '01-introduction'}}
```

**④ 画像・動画パスの修正**

```
/assets/ファイル名（画像・動画）
↓
../images/assets/ファイル名
```

- 対象拡張子: `.png`, `.jpg`, `.jpeg`, `.gif`, `.webp`, `.mov`, `.mp4` など
- 行頭に生パスとして残っているケース（`/assets/xxx.mov` のみの行）も同様に変換する

**⑤ ヘッダー余分スペースの除去**

Note.com エクスポート特有の `## ` や `### ` 後の余分なスペースを除去する:

```
##  タイトル  →  ## タイトル
###  タイトル  →  ### タイトル
```

ヘッダー記号直後のスペースが2つ以上の場合、1つに統一する。

**⑥ 空ヘッダー行の削除**

コンテンツを持たないヘッダー行（`##` や `###` のみで本文がない行）を削除する:

```
##
##   （スペースのみ）
```

→ 行ごと削除する（前後の空行も整理する）。

**⑦ エスケープ済み Obsidian スタイルリンクの削除**

Note.com から生成した Markdown に含まれる `!\[\[...\]\]` 形式（バックスラッシュでエスケープされた Obsidian wiki リンク）を削除する:

```
!\[\[filename.png\]\]
```

→ 行ごと削除する。対応する画像が `images/` に存在する場合は `![](../images/filename.png)` に変換してもよい。

**⑧ リスト内「1.  → 」ネストの整形**

Note.com エクスポート特有の、箇条書きの中にネストされた番号付きサブリストを整形する:

```markdown
-   親項目

    1.  → 説明内容

```

→ `1.  → ` のプレフィックスを除去してフラットな箇条書きにまとめるか、親項目の説明として自然につなげる:

```markdown
-   親項目 → 説明内容
```

**⑨ 箇条書きの余分スペース・空行の整理**

Note.com エクスポート特有の `-   `（ハイフン＋複数スペース）と、リスト項目間の余分な空行を整理する。

**余分スペースの除去:**

```markdown
<!-- 変換前 -->
-   黒・白背景で最も目立つ
-   緑・黄色背景ではかなり見えにくい

<!-- 変換後 -->
- 黒・白背景で最も目立つ
- 緑・黄色背景ではかなり見えにくい
```

- `-` の直後のスペースが2つ以上の場合、1つに統一する（ネスト箇条書きの先頭インデントは保持）
- スペースのみの行（`    ` など）は空行（完全に空の行）に変換する

**リスト項目間の空行の除去:**

```markdown
<!-- 変換前 -->
- 黒・白背景で最も目立つ

- 緑・黄色背景ではかなり見えにくい

<!-- 変換後 -->
- 黒・白背景で最も目立つ
- 緑・黄色背景ではかなり見えにくい
```

- 前後どちらもリスト項目（`- `・`* `・`1. ` 等）の場合のみ間の空行を全て除去する
- リスト項目と通常段落の間の空行は除去しない

**連続空行の圧縮:**

- コードブロック外で2行以上連続する空行は1行に圧縮する

**⑩ ヘッダーの手動行番号の除去**

Note.com エクスポートで付いた手動の連番プレフィックスをヘッダーから除去する。

除去対象のパターン:

```
## 1\. タイトル    →  ## タイトル
### 2\. タイトル   →  ### タイトル
### 1.1 タイトル   →  ### タイトル
### 4-1. タイトル  →  ### タイトル
## 0\. タイトル    →  ## タイトル
```

- `数字\.`（エスケープあり）・`数字.`（エスケープなし）・`数字-数字.`（サブ番号）いずれも除去する
- ヘッダー記号（`#`）の直後にある番号プレフィックスのみが対象（本文中の数字は変更しない）
- 番号の直後のスペースが通常スペース・ノーブレークスペース（`\u00A0`）どちらの場合も除去する
- ヘッダー自体のテキスト内容（例: `1からの導入方法`・`4Stepに実行を制限`）は変更しない

**⑪ 画像サイズの統一と Alt Text 設定**

Note.com エクスポートの画像タグは `height` 指定や Alt Text なしのことが多いため、横幅固定・Alt Text 付きに統一する。

```markdown
<!-- 変換前 -->
![,height:250px](../images/assets/xxx.png)
![](../images/assets/xxx.png)

<!-- 変換後 -->
![画像の内容を表す説明文,width:300px](../images/assets/xxx.png)
```

- `height:Xpx` → `width:300px` に置き換える
- Alt Text が空（`,height:` の前が空、または `![]`）の場合は、前後の文脈（見出し・説明文）から適切な説明文を推測して設定する
- すでに `width:` が指定されている場合は変更しない
- すでに Alt Text が設定されている場合は変更しない（サイズ指定のみ変換する）

**⑫ リンク付きヘッダーのQRコード変換**

見出し（`#`〜`######`）に外部リンクが含まれている場合、リンクを除去してヘッダーをプレーンテキストにし、リンク先 URL を直後行に QRコードとして追加する。

```markdown
<!-- 変換前 -->
### [Z-Image Turbo](https://ollama.com/x/z-image-turbo)

<!-- 変換後 -->
### Z-Image Turbo

<!-- qrcode: https://ollama.com/x/z-image-turbo -->
```

- ヘッダー行の `[テキスト](URL)` を `テキスト` のみに置き換える
- ヘッダー行の直後に空行を入れてから `<!-- qrcode: URL -->` を追加する
- 対象: `http://` / `https://` の外部 URL のみ
- ローカルホスト URL（`localhost`・`127.0.0.1`・LAN ホストなど）は除外する

**⑬ リンクとQRコードの重複削除**

同じ URL のスタンドアロンリンクと QRコードが連続している場合、リンク行を削除して QRコードのみ残す。

```markdown
<!-- 変換前 -->
[https://ollama.com/](https://ollama.com/)

<!-- qrcode: https://ollama.com/ -->

<!-- 変換後 -->
<!-- qrcode: https://ollama.com/ -->
```

- 判定条件: リンクテキストと href が同一 URL（末尾スラッシュ・バックスラッシュエスケープの差異は無視）
- リンク行と直後の空行を削除し、QRコード行はそのまま残す
- リンクがリスト項目（`-   [URL](URL)`）の場合も同様にリスト項目ごと削除する
- 前方5行以内に同一 URL の `<!-- qrcode: ... -->` がある場合を対象とする

**⑭ 外部 URL へのQRコード埋め込み**

読者がスマートフォンなどでアクセスしやすいよう、外部リンクの直後に `<!-- qrcode: URL -->` を追加する。

```markdown
[リンクテキスト](https://example.com)

<!-- qrcode: https://example.com -->
```

**対象とするリンク:**

- 1行全体が外部リンクになっている行（スタンドアロンリンク）
- 読者が実際に訪問する可能性のあるページ（ツール配布ページ・モデルダウンロードページ・公式ドキュメントなど）

**除外するリンク:**

- コードブロック内の URL（`curl`、`git clone` などのコマンド中の URL）
- `http://127.0.0.1` や `http://localhost` などのローカルホスト URL
- LAN 内ホスト URL（例: `http://macbookpro.lan:7860`）
- 直接ファイルダウンロード URL（`.safetensors`、`.onnx`、`.gguf` などのバイナリ）
- 同一 URL が複数回登場する場合は初出のみ（重複は追加しない）
- 文章中にインラインで埋め込まれたリンク（行全体がリンクでないもの）

### description の確認

自動生成した `description` を提示してユーザーの承認を得る。修正が必要な場合は修正してから書き込む。

### ファイルの書き込み

- ユーザーの承認後、変換済み内容をファイルに書き込む
- 書き込み先: 元のファイルパスを上書き（または `docs/` に新規作成）
- ユーザーが別パスへの出力を希望する場合はそれに従う

### 後続作業の案内

書き込み完了後、以下を案内する:

1. **番号重複チェック**: `docs/` ディレクトリのファイル一覧を確認し、番号が重複していないかチェックする
2. **プレビュー確認**: `npm run dev` でブラウザプレビューを起動して表示を確認する
3. **TODO の対応**: `TODO: chapref` が残っている場合は guid と docs ファイル名の対応を確認して置き換える

## フェーズ4（任意）: 索引タグの追加

全章が揃ったタイミングで、索引ページ（Appendix）に掲載すべき用語に索引タグを付与する。

### 記法

```markdown
{{appendix filePath 'anchor-id' '用語名'}}
```

- `filePath` はクォートなし（Handlebars テンプレート変数）
- `anchor-id`: ファイル内で一意のID（英小文字・ハイフン、例: `stable-diffusion`）
- `用語名`: 索引に表示する文字列

コードブロック・表・画像 Alt Text など Markdown 記法の中に用語がある場合は、その要素の直前か直後に hidden 版を置く:

```markdown
{{appendix-hidden filePath 'anchor-id' '用語名'}}
```

### 配置ルール

| 場所 | 使う記法 | 配置位置 |
|------|---------|---------|
| 通常テキスト（本文） | `{{appendix ...}}` | 用語のある文中・行末に**用語テキストの代わりに**置く |
| ヘッダー（`#` `##` `###` など） | `{{appendix-hidden ...}}` | ヘッダー行の**直後の行**（ヘッダー行内には入れない） |
| コードブロック内の用語 | `{{appendix-hidden ...}}` | コードブロックの直前または直後の行 |
| 画像の Alt Text 内の用語 | `{{appendix-hidden ...}}` | 画像タグの直後の行 |
| 表内の用語 | `{{appendix-hidden ...}}` | 表の直前または直後の行 |

> **重要: `{{appendix}}` タグは用語テキストをそのまま表示する。用語を二重に書かないこと。**
>
> ```markdown
> NG: mini-LED{{appendix filePath 'mini-led' 'mini-LED'}}  ← "mini-LED" が2回表示される
> OK: {{appendix filePath 'mini-led' 'mini-LED'}}          ← タグが "mini-LED" を表示する
> ```
>
> 文章の流れを保つため、用語が含まれる文の該当箇所をタグで置き換える:
> ```markdown
> NG: MacBook Proのmini-LED{{appendix filePath 'mini-led' 'mini-LED'}}パネルは...
> OK: MacBook Proの{{appendix filePath 'mini-led' 'mini-LED'}}パネルは...
> ```

- 各ファイルで同じ anchor-id は1回だけ使う（重複不可）
- 同じ用語が複数ファイルに登場する場合はそれぞれのファイルに追加する
- 各ファイルで**初出の目立つ箇所**のみタグを付ける（全出現箇所に付けない）

### 索引化すべき用語の目安

- ソフトウェア・ツール名（Stable Diffusion, ComfyUI, Ollama, Homebrew など）
- AIモデル名（LoRA, VAE, ControlNet, ADetailer, Z Image Turbo など）
- 技術用語・略語（MPS, PyTorch, BF16, VRAM, RGBA, Wake on LAN など）
- ハードウェア固有名（Apple Silicon, M1 Max, MacBook Pro, Mac Studio など）
- 本書の主要概念（クラムシェル, インペイント, Civitai など）

### 例

```markdown
## Homebrewとは？
{{appendix-hidden filePath 'homebrew' 'Homebrew'}}

{{appendix filePath 'homebrew' 'Homebrew'}}はmacOS用パッケージマネージャーです。

{{appendix filePath 'lora' 'LoRA'}}はモデルを軽量化する仕組みです。

![Apple Metalを選ぶ,width:300px](../images/assets/xxx.png)

## フェーズ5（任意）: 技術書典 コンテンツポリシー対応

技術書典の審査で R-18 相当コンテンツを含むとして否承認された場合に実施する修正フェーズ。

技術書典は全年齢対象のイベントであるため、成人向けコンテンツに関する表現を以下の2方針で修正する。

---

### 方針A：ツール・モデルの正式名称 → 伏字（●●●●）に置換

コマンドやコードブロック内も含め、固有名称内の成人向け語句を `●●●●` に置換する。**URLはそのまま維持する（機能に必要なため）。**

| 置換前 | 置換後 |
|--------|--------|
| `WAI-NSFW-illustrious-character-select` | `WAI-●●●●-illustrious-character-select` |
| `huihui_ai/gpt-oss-abliterated` | `huihui_ai/gpt-oss-●●●●●●●●●` |
| `animeNSFWDetection_v50Variant1.zip` | `anime●●●●Detection_v50Variant1.zip` |
| `waiNSFW`（設定値内など） | `wai●●●●` |

---

### 方針B：説明文・セクション・プロンプト例 → 削除して違和感のない形に修正

成人向けコンテンツを示す説明・タイトル・例示プロンプトを削除し、技術的な流れが自然に続くよう書き換える。

**削除・置換対象の種別と対応:**

| 種別 | 対応 |
|------|------|
| 「NSFW」「えっち」を含むセクションタイトル | 語句を削除するか全年齢向けタイトルに変更 |
| 「規制解除（Uncensored）」の機能説明 | 「テキストエンコーダーを変更することで画風を変化させる」等のSFWな説明に変更 |
| 性的示唆プロンプト語句（"seductive"/"alluring"/"show me your pants"等） | SFWなプロンプト（"cheerful"/"bright eyes"/"smile"等）に差し替え |
| キャラクター掛け合いの「規制解除のモデルがあるの？」等のセリフ | 内容に合わせたSFWなセリフに差し替え |
| 「ちょっとえっちすぎる絵がでました」等のコメント | 削除 |
| 「審査通るように黒塗り入れてます」等の注記 | 削除（審査への言及自体が問題のため） |
| NSFW対応モデル追加セクション全体 | セクションごと削除し前後の流れを整える |
| 説明文・目次の「NSFWな…」の語句 | 語句を削除またはSFWな表現に変更 |

---

### 確認キーワード（修正後に残漏れチェック）

修正後に以下のコマンドで残漏れを確認する:

```bash
grep -r "NSFW\|えっち\|abliterated\|規制解除\|色気" docs/
```

**残っていても問題ないケース:**

- GitHub リポジトリの URL（例: `https://github.com/lanner0403/WAI-NSFW-illustrious-character-select-EN`）
- 方針Aで伏字済みの名称（例: `WAI-●●●●-illustrious-character-select`）

---

### 修正手順

1. 対象ファイルをすべて読み込んで成人向けコンテンツの箇所を洗い出す
2. 方針AとBを適用して各ファイルを修正する（方針の使い分けは上記表を参照）
3. `grep` で残漏れを確認する
4. コミット・プッシュする

**コミットメッセージ例:**

```
技術書典向けにR-18相当コンテンツを全年齢対象に修正

- ツール・モデルの正式名称内のNSFW/abliterated語句を伏字（●●●●）に置換
- NSFW生成・規制解除・成人向けモデルの説明文を削除またはSFWな内容に差し替え
- 性的示唆プロンプトをSFWプロンプトに差し替え
- 成人向けセクション（NSFW対応モデルの追加等）を削除
```
